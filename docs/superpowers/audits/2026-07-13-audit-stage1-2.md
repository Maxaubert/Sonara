### [IMPORTANT] src/sonara/chatterbox.py:267 - A spawn failure raises a non-ChatterboxError that bypasses the Kokoro fallback, silently dropping every chunk of the utterance

_spawn's subprocess.Popen (chatterbox.py:267) and chatterbox_venv_python() are unguarded, so if the venv python disappears mid-run or Popen fails (OSError/FileNotFoundError), _request -> _ensure_proc -> _spawn propagates a raw OSError out of synth_wav. The daemon-side per-chunk fallback catches ONLY chatterbox.ChatterboxError (platform/windows/tts.py:493), so the OSError escapes it and is swallowed by the pipelined producer's blanket guard (tts.py:230-233, 'wav = None ... continue'). Result: no Kokoro fallback, no fallback notice, and the entire utterance is silently skipped chunk by chunk, defeating the stated 'the stream never goes silent' contract. is_provisioned() is only checked once up front (tts.py:523), so this window is real.

**Fix:** Wrap the Popen call (or the whole _spawn body) in try/except and re-raise as ChatterboxError, so every failure mode of the client funnels through the one exception type the fallback path handles.

**Verifier:** Confirmed. _spawn's subprocess.Popen (chatterbox.py:267) is unguarded and chatterbox_venv_python() is a pure path join ("may not exist" per its docstring), so a missing venv python or any Popen OSError propagates raw through _ensure_proc/_request (including the mid-request respawn at chatterbox.py:337) out of synth_wav. The daemon's per-chunk fallback (tts.py:493) catches only ChatterboxError, so the OSError escapes into _produce's blanket guard (tts.py:230-233) which sets wav=None and skips the chunk: no Kokoro fallback, no fallback notice. It is actually worse than stated: the failed spawn leaves _proc stale so every chunk re-raises, yet the producer still enqueues _DONE and the consumer returns rc=0, so a fully silent utterance reports success. Two docstrings ("the stream never goes silent", "synth_one owns its own fallback") show this is contract-breaking, not intentional. Mitigation is only partial: is_provisioned() is re-checked per utterance (tts.py:523), so a deleted venv silently drops just the in-flight utterance, but non-ENOENT Popen failures (access denied, handle/memory exhaustion) pass that check and would silently drop every Chatterbox utterance for as long as the condition persists. daemon.py:929's warm() call has the same unhandled-OSError hole. Severity "important" is correct.

### [IMPORTANT] src/sonara/chatterbox.py:329 - No failure memo/backoff: a persistently broken worker is respawned twice per chunk for every chunk, stalling each chunk up to 2x chatterbox_timeout before its Kokoro fallback

_request (chatterbox.py:329-342) does kill -> respawn -> retry on every call independently. If the worker reliably dies or times out (e.g. cold model load slower than chatterbox_timeout on a slow disk, recurring CUDA init crash), then EVERY chunk of EVERY utterance pays the full kill/respawn/retry cycle: up to 2x120 s of dead time per chunk with the default timeout before tts.py's per-chunk fallback finally speaks it via Kokoro. A multi-chunk digest turns into minutes of intermittent silence, and the GPU is thrashed with repeated partial model loads. There is no circuit breaker or 'declare it down for this utterance/N seconds' state.

**Fix:** After a respawn-retry failure (the 'chatterbox worker is unavailable' raise at line 341), record a cooldown timestamp on the client; while inside the cooldown, synth_wav raises ChatterboxError immediately so the per-chunk Kokoro fallback fires fast. Clear the cooldown on the next successful request or after N seconds.

**Verifier:** CONFIRMED with one correction. The core debt is real: ChatterboxClient._request (src/sonara/chatterbox.py:329-342) has no failure memo, cooldown, or circuit breaker (grep for cooldown/circuit/backoff across src/sonara finds nothing), every failed chunk leaves _proc killed, and the next chunk's _ensure_proc respawns and repays the full cost. The per-chunk Kokoro fallback (platform/windows/tts.py:490-498) only rescues the single chunk; nothing stops routing subsequent chunks or utterances back to the dead worker (warm() is best-effort and does not gate routing; gate_ok checks VRAM only). The consumer loop tolerates arbitrary producer stalls (queue.Empty poll at tts.py:269-271), so an N-chunk digest with a hung worker really does stall ~N x chatterbox_timeout (default 120 s each) with intermittent silence. Correction: the '2x timeout per chunk' claim is overstated - _try_request RAISES ChatterboxError("timeout") on a stuck read (line 319), which propagates out of _request without the respawn-retry; the kill/respawn/retry (2 spawns per chunk) only runs on dead-pipe failures (write error or empty read), which fail in crash-time, not timeout-time. Worst case per chunk is ~1x timeout plus a fast crash cycle. Severity 'important' still fits the corrected impact (minutes of stall per digest, GPU thrash, but speech never fully lost).

### [MINOR] src/sonara/platform/windows/tts.py:289 - _ChatterboxHandle: terminate() landing between the post-get abort check and the _cur_sub assignment lets the just-started chunk play to completion despite the cancel

In wait() (tts.py:272-289) the consumer checks self._abort after q.get(), then runs _on_play (audio ducking, which iterates COM audio sessions and can take tens of ms on the first chunk), then calls self._play(item) which STARTS winsound playback, and only then assigns self._cur_sub = sub. terminate() (tts.py:321-328) sets _abort and terminates self._cur_sub, but if it fires inside that window _cur_sub is still None, so nothing stops the newly started sub-handle. The consumer then blocks in sub.wait(timeout=60) (tts.py:291), which only returns when the sub's duration timer fires, so a cancelled/muted utterance keeps speaking one full chunk (up to 280 chars, roughly 15-25 s of audio) after the user cancelled. The abort is only noticed at the top of the next loop iteration.

**Fix:** After self._cur_sub = sub, re-check the abort flag: if self._abort.is_set(): sub.terminate(). That closes the window because terminate() has already set _abort before it reads _cur_sub.

**Verifier:** Confirmed by reading tts.py. The consumer's last abort check is at line 272; playback starts at line 284 (_play -> winsound SND_ASYNC) and _cur_sub is only assigned at line 289. terminate() (321-328) sets _abort then terminates _cur_sub, which is None in that window (reset at 297), so a cancel landing there stops nothing. _TtsHandle.wait() blocks on an Event set only by a duration Timer or its own terminate(), which is never called on the racing sub; wait()'s finally block never terminates _cur_sub, and speaker.py's cancel() calls terminate() exactly once with no global PlaySound(None,0) fallback. The chunk therefore plays to completion (up to ~15-25 s) after the cancel. The proposed post-assignment abort re-check is a correct close of the window. However, the window is only a few ms per chunk (tens of ms on the first chunk due to ducking), and the common cancel timing - during the multi-second synthesis stall - is caught by the q.get abort checks; once audio is audible _cur_sub is already set and terminate works. Real but low-probability, so downgraded to minor.
